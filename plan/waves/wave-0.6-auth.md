# موجة 0.6 — الحماية والتحقق

> الحالة: مكتملة مبدئياً — 24 أبريل 2026

## الهدف

إغلاق أساس الحماية قبل التوسع في الخزينة والتحصيل، بحيث لا يبقى النظام مفتوحاً أو قابلاً لإدخال بيانات غير مضبوطة من الواجهة أو من API مباشر.

## المنجز

- تسجيل دخول حقيقي عبر جدول `users` في PostgreSQL.
- جلسات عبر JWT داخل HttpOnly cookie للويب.
- دعم Bearer token لتطبيق الموظفين.
- حماية مسارات `stations/employees/tasks/fuel/treasury/users/attendance`.
- Rate limiting على `/api/auth/login`.
- سجل تدقيق للعمليات الحساسة في الدخول، المستخدمين، المحطات، الديزل، والخزينة.
- حصر حسابات الدخول حالياً على مدير النظام فقط حسب قرار المستخدم.
- تقييد البيانات حسب الدور والمحطة في الوحدات الأساسية.
- إعدادات cookie آمنة وقابلة للضبط عند النشر.
- اختبار 401/403/400 لمسارات API الأساسية.

## ملاحظة انتقالية

قبل إعادة تفعيل حسابات المدراء والفنيين عملياً، يجب مراجعة ربط كل مستخدم بمحطته وموظفه من شاشة المستخدمين أو seed حقيقي حتى تعمل القيود حسب الواقع.

## قاعدة العمل

أي ميزة جديدة بعد هذه النقطة يجب أن تدخل من خلال:

1. endpoint محمي بـ `requireAuth`.
2. validation واضح للمدخلات.
3. رسالة خطأ عربية.
4. سجل تدقيق إذا كانت العملية تغيّر بيانات مالية، صلاحيات، حسابات، أو مواقع.
